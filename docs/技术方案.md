# 技术方案

本文说明“司机画像”静态 CDN 版的实现方案、数据构建流程、部署方式和主要取舍。

## 1. 总体架构

当前方案采用多文件静态站点：

```text
每日 Excel
  -> scripts/import-driver-data.py
  -> dist/data/drivers.json 日期索引
  -> dist/data/daily/drivers-YYYY-MM-DD.json 按日明细
  -> dist/data/filter-options.json
  -> dist/data/strategy-thresholds.json
  -> dist/data/manifest.json

reference/current-cdn-frontend/
  -> scripts/prepare-static-site.py
  -> dist/index.html
  -> dist/app.<build-id>.js
  -> dist/styles.<build-id>.css

dist/
  -> scripts/cdn-dir-deploy.py
  -> 公司内网 CDN
```

前端不依赖后端服务。用户打开 CDN 链接后，浏览器读取同目录下的静态 JSON 文件完成查询、筛选、列表、详情、建议策略和 CSV 下载。

## 2. 为什么采用多文件静态版

曾经考虑过把所有数据内联到单个 HTML，或只上传一个 HTML 文件。但当前项目数据量较大，并且 CDN 一键部署包有 800MB 上限，单文件内联会带来几个问题：

- HTML 文件过大，打开慢且容易部署失败。
- 每次访问都要下载过多内容。
- 更新数据时容易只覆盖当天数据，导致窗口内旧日期丢失。
- 多日数据混在一个 `drivers.json` 中，数据日期筛选容易失效。

因此当前稳定方案是：

- `index.html` 只负责页面入口。
- `app.<build-id>.js` 和 `styles.<build-id>.css` 使用构建时间文件名，减少缓存问题。
- `drivers.json` 只做日期索引。
- 每天司机明细放在独立 `daily/drivers-YYYY-MM-DD.json`。
- 切换数据日期时加载对应日期文件。

## 3. 前端加载流程

```text
打开 index.html
  -> 加载 app.<build-id>.js 和 styles.<build-id>.css
  -> 读取 data/manifest.json
  -> 展示静态站点生成时间和数据窗口
  -> 读取 data/filter-options.json
  -> 读取 data/drivers.json 日期索引
  -> 默认加载 latestDate 对应 daily 文件
  -> 用户查询或筛选
  -> 必要时加载选中日期 daily 文件
  -> 前端在已加载数据中完成筛选、列表和详情展示
```

页面加载静态数据时显示进度条。加载完成后，用户可以：

- 输入司机 ID 查询。
- 选择城市、公司、产品线和数据日期。
- 使用高级筛选。
- 点击左侧司机列表切换右侧详情。
- 下载当前筛选结果 CSV。

## 4. 数据构建流程

入口脚本：

```bash
python3 scripts/import-driver-data.py <输入文件> --upload-date YYYY-MM-DD --out-dir dist/data
```

支持输入：

- `.xlsx`
- `.xls`
- `.csv`
- `.json`

主要步骤：

```text
读取源文件
  -> 中文列名映射成标准字段
  -> 清洗司机 ID、数字、日期
  -> 推断或读取 upload_date
  -> 计算 window_start = upload_date - 6 天
  -> 过滤窗口内数据
  -> 迁移旧版 drivers.json 明细到 daily 文件
  -> 删除窗口外 daily 文件
  -> 只覆盖本次输入日期的 daily 文件
  -> 完整解码一次窗口内所有 daily 文件
  -> 生成 DatasetSummary（行数、日期、每日文件、筛选项）
  -> 复用摘要生成 drivers.json / filter-options.json / meta.json
  -> 计算建议策略和 strategy-thresholds.json
  -> 写入 manifest.json
  -> 顺序重建 filter-index 和 driver-lookup 派生索引
  -> 检查整个 dist 大小不超过 800MB
  -> 复用摘要回写最终包体元信息
```

导入性能快速路径：

- 正常发布包未超过 `800MB` 时，Python 主流程只完整解码一次近七天 daily 文件。
- `drivers.json`、`filter-options.json`、`manifest.json` 和 `meta.json` 都复用同一份 `DatasetSummary`；最终包体数字回写只改小型元信息文件。
- 组合筛选索引和司机 ID 直达索引的构建器、gzip 格式和前端查询契约不变，不以页面查询性能换取导入速度。
- 只有整个 `dist/` 超过 `800MB` 时，才在删除最旧日期后重新解码剩余日文件并重建两套派生索引。
- 真实数据基准：2026-07-15 完整导入约 `154 秒`，2026-07-16 约 `181.47 秒`，2026-07-17 的 `903260` 条快照、`589.955MB` 发布包完整导入约 `214.34 秒`。具体时间会随 Excel 解码、机器和磁盘环境浮动。

关键约束：

- `drivers.json` 不能放多天司机明细。
- `daily/drivers-YYYY-MM-DD.json` 只能放该自然日司机明细。
- 导入新一天数据时，要保留仍在 7 天窗口内的旧日期文件。
- 如果最近 7 天产物超过 800MB，脚本先依靠字段裁剪和紧凑 JSON 降低体积；仍超过时，按日期从旧到新剔除并重建索引和筛选项。
- 每次因体积剔除日期，都要写入 `manifest.actual_dates`、`manifest.dropped_dates`、`manifest.drop_reason` 和 `manifest.package_size_mb`。
- 如果最新 1 天单独仍超过 800MB，脚本停止发布，后续应升级为按城市/分片加载或后端 API。
- 如果源文件没有 `dt` 或可解析日期，数据日期功能无法可靠工作，应停止并提示补充。

## 5. 建议策略计算

策略配置文件：

```text
config/strategy-rules.json
```

计算层：

```text
scripts/import-driver-data.py
```

当前生产阈值策略：

```text
policy = case_mean
operator = 按策略配置，默认 >
driver_metric 与 death_case_reference[reference_metric] 的均值比较
```

策略计算在数据导入阶段完成，输出到每条司机记录的 `strategyKeys` 和少量 `strategyEvidence`。前端只负责按 `config/strategy-rules.json` 展示标题、优先级、标签和话术转译，不重新写死判断逻辑。

话术转译 V1 也放在 `config/strategy-rules.json` 中，每条规则通过 `translation.driver_script`、`translation.action_advice`、`translation.communication_tip` 配置。页面主显司机友好话术和可操作建议，原始内部依据默认折叠在“参考依据”里，复制按钮只复制对外部分。

建议策略展示顺序按标签梯度优先、`priority` 其次：高风险+可解释优先，其次仅高风险，其次仅可解释，最后无标签；同一梯度内按 `priority` 升序。高风险标签显示为红色，可解释标签显示为黄色。具体字段口径见 [建议策略优先级与标签口径](建议策略优先级与标签口径.md)。

`past_7_day_non_listening_period` 只保留为司机资料背景字段，不再作为建议策略触发字段。它更像出车节奏或在线行为，不适合作为健康风险建议的直接依据。

这样做的好处是：

- 后续更换阈值算法时，只改配置和计算层。
- 页面不会散落阈值数字、策略文案和话术文案。
- 缺字段、样本不足、参考字段缺失等情况可以在阈值产物中排查。

无策略命中时，导入脚本输出 `fallbackRule` 对应的 `常规关怀`。

## 6. 前端构建流程

入口脚本：

```bash
python3 scripts/prepare-static-site.py --out-dir dist
```

脚本从：

```text
reference/current-cdn-frontend/
```

复制：

```text
index.html
app.js
styles.css
```

并生成带构建时间的文件：

```text
dist/app.<build-id>.js
dist/styles.<build-id>.css
```

然后把 `dist/index.html` 中的引用替换成带 build-id 的文件名，并删除旧的 `index-inline.html`。当前 CDN 静态版不使用 `index-inline.html`。

## 7. CDN 部署方案

部署脚本：

```text
scripts/cdn-dir-deploy.py
```

这个脚本是基于原先 `cdn-deploy` 单文件/HTML 发布思路修改而来。当前项目需要的是目录递归上传，因此把发布口径改成：

```text
递归上传整个 dist/
保留相对路径
返回 CDN index.html 直链
```

部署命令：

```bash
python3 scripts/cdn-dir-deploy.py --dir dist --deploy-id driver-profile-YYYYMMDD-HHMM
```

脚本会上传 `dist/` 下所有文件，并输出：

- CDN base URL。
- CDN `index.html` 直链。
- `data/` 路径。
- SmartWork app URL 提醒。

注意：SmartWork app URL 可能是包装页，不适合多文件静态站点验收。最终应使用 CDN 的 `index.html` 直链。

## 8. CDN 路径和缓存

推荐每日更新时使用新的 `deploy-id`：

```text
driver-profile-YYYYMMDD-HHMM
```

这样可以避免：

- 浏览器缓存旧 `index.html`。
- CDN 缓存旧 `app.js`。
- 同事看到旧版本页面，以为新功能没有部署。

如果公司内部必须沿用旧链接，并且页面代码没有变化，可以只递归覆盖旧路径下的 `dist/data/`。但这要求部署工具能保留 `data/daily/` 子目录结构。

## 9. 本地预览和验收

本地启动：

```bash
python3 -m http.server 8000 --directory dist
```

访问：

```text
http://localhost:8000/index.html
```

建议检查：

- 页面标题为 `司机画像`。
- 加载静态数据时有进度条。
- 左侧展示 `静态站点生成时间`。
- 高级筛选包含 `是否组织化`。
- 数据日期切换可用。
- 城市筛选能展示司机列表。
- 点击司机能展示综合画像、建议策略和司机资料。
- 下载 CSV 可用。


## 10. 安全和数据边界

- 文档和代码示例不得写入真实司机数据。
- 生产 Excel 只作为构建输入，不提交到公开仓库。
- 静态 JSON 包含可查询司机数据，必须只发布到受控 CDN。
- `drivers.json` 不得塞入多天司机明细。
- `daily` 文件不得包含窗口外司机明细或页面不需要的全量原始字段。
- CSV 不得导出全量原始字段。

## 11. 主要取舍

| 方案 | 优点 | 代价 |
|---|---|---|
| 多文件静态 CDN | 部署简单，无后端依赖，查询在浏览器完成 | 首次加载某日大文件会消耗访问者流量 |
| 按日独立 daily 文件 | 日期功能可靠，更新当天不误删旧日期 | 切换日期时需要再加载对应文件 |
| 超 800MB 时保留最近可容纳日期 | 不硬传超限包，仍保证最新数据可用 | 实际可查日期可能少于默认 7 天，必须在 manifest 中说明 |
| 新 deploy-id | 缓存问题少，最适合 agent 一键部署 | 每天会产生新链接，需要同步给同事 |
| 沿用旧链接覆盖 data | 链接稳定 | 更容易遇到缓存和漏文件问题 |

## 12. 后续可演进方向

- 当数据规模继续增长时，把 `daily` 文件再按城市、公司或分片拆分。
- 增加公司内部 API 后端，让浏览器只拉取筛选后的结果。
- 增加固定发布流水线，减少依赖 agent 人工执行。
- 增加访问权限、审计和数据脱敏策略。
- 把策略阈值算法从 `case_mean` 切换为新的配置化算法。
