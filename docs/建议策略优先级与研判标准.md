# 建议策略优先级与研判标准

> 当前口径日期：2026-07-21  
> 策略配置版本：`2026-07-09`  
> 适用范围：风险前哨静态站点的建议策略触发、卡片排序和内部解释

## 1. 文档定位

本文集中说明当前生产版的四件事：策略如何命中、case 均值如何计算、命中策略如何排序、缺失数据如何处理。

权威实现来源按以下顺序理解：

1. `config/strategy-rules.json`：规则、字段映射、标签、P0-P3 和数字优先级。
2. `scripts/import-driver-data.py`：case 均值计算、条件判断和命中 key 生成。
3. `reference/current-cdn-frontend/app.js`：页面上的最终策略卡片排序。
4. `dist/data/strategy-thresholds.json`：本次发布实际生成的阈值、样本数和不可用原因。

本文是当前实现的解释，不替代上述配置和生成产物。规则或参考样本变化后，应重新生成阈值并同步更新本文。

## 2. 研判边界

- 建议策略用于客服关怀优先级和沟通准备，不是医学诊断、事故归因或处罚依据。
- `riskTierRank` 来自上游模型；建议策略不会重新计算司机排名。
- case 均值和具体阈值属于内部依据，只能出现在默认折叠的内部信息中，不得进入对司机话术或复制内容。
- 没有策略命中时展示 `常规关怀`，其优先级为 `999`。

## 3. case 均值研判

### 3.1 计算方式

对每个 `reference_metric`，读取死亡 case 参考文件中的有效有限数值：

```text
case_mean(metric) = 有效样本值总和 / 有效样本数
```

司机数值指标按配置中的运算符比较：

```text
正向指标：driver_metric > case_mean(reference_metric)
反向指标：driver_metric < case_mean(reference_metric)
```

当前反向指标是 `min_sleep_duration`：最短睡眠时长低于 case 均值时命中。其余已配置数值指标主要使用 `>`。

规则包含 `condition.any` 时，任意一个子条件命中即可触发该策略；包含 `condition.all` 时，所有子条件都命中才触发。

### 3.2 缺失处理

以下任一情况出现时，对应数值条件不命中：

- 司机指标缺失、空白或不能转成有限数字。
- 参考字段不存在或没有有效样本。
- case 均值无法生成。
- 规则被设置为 `enabled: false`。

不使用页面兜底阈值，也不会因为城市、产品线或宽泛健康标签自动补命中。

### 3.3 布尔替代条件

当前数值阈值统一使用 `case_mean`，但部分规则还配置了布尔替代字段，取值属于 `是 / 1 / true / TRUE / Y / y` 时可直接命中：

| 字段 | 用途 |
|---|---|
| `is_long_consecutive` | 长期连续出车替代信号 |
| `is_sleep_deprived` | 睡眠不足替代信号 |
| `is_sudden_fatigue` | 突发疲劳替代信号 |
| `is_regular_night` | 常规夜班替代信号 |

因此，严格描述应是：**生产数值阈值采用 case 均值；已配置的布尔替代字段可作为直接命中条件。** 不能笼统表述为“所有策略都必须经过 case 均值比较”。

## 5. 优先级与页面排序

页面不是单纯按 P0-P3 或数字 `priority` 排序。实际排序依次比较：

1. 同时有 `高风险` 和 `可解释` 标签。
2. 只有 `高风险` 标签。
3. 只有 `可解释` 标签。
4. 无上述标签。
5. 同一标签组内，`priority` 数字越小越靠前。
6. 数字也相同时，按策略 key 排序。

因此标签组优先于数字。例如 `service-duration-high` 的数字优先级为 `80`，但它同时具有高风险和可解释标签，所以会排在只有高风险标签、数字优先级为 `70` 的 `service-duration-7d-high` 前面。

## 6. P0-P3 的业务含义

- `P0`：核心优先风险，风险显著且容易向业务解释，原则上优先沟通。
- `P1`：重要风险或核心风险的替代指标，用于补充 P0 研判。
- `P2`：解释性补充信号，适合作为沟通背景，不应压过高风险策略。
- `P3`：背景型解释信息，如年龄；不能单独等同于健康异常。

P级是配置中的业务分层标识；当前页面最终顺序仍以“标签组 → 数字 priority → key”为准。

## 7. 当前实现差异与治理项

### 最小参考样本数

配置声明 `min_sample_count: 30`，但当前计算代码实际只判断有效样本数是否大于 `0`，尚未强制执行 30 条门槛。因此当前文档不能声称“少于 30 条一定不触发”。如果要严格执行，应单独修改计算层并增加回归测试。

### 无阈值规则

疲劳评分和身体指标策略虽然处于启用状态，但其参考字段当前样本数为 `0`。在补齐参考数据前，这些数值条件不会命中；不得使用人工阈值替代。

### 口径命名

`lately_7d_except_sub_online_dur_hour` 是近7日非预约单在线时长，不是近7日服务时长。页面画像已按“近7天日均在线时长”展示；策略标题中的“近7天服务时长偏高”属于现有业务标题，内部依据仍应明确实际命中字段。

## 8. 变更与验收要求

修改参考样本、规则字段、case 均值算法、标签、P级或 priority 后，必须：

1. 更新 `config/strategy-rules.json`，不得把阈值写进页面代码。
2. 重新运行数据导入，生成新的 `strategy-thresholds.json` 和 daily 策略 key。
3. 检查阈值、样本数、缺失原因和启用状态。
4. 验证策略卡片顺序满足“高风险+可解释 → 高风险 → 可解释 → 无标签”。
5. 验证无命中司机只显示 `常规关怀`。
6. 验证对外话术和复制内容不含 case 均值、死亡 case、阈值数字或医学结论。
7. 更新本文的口径日期、配置版本和当前阈值表。
