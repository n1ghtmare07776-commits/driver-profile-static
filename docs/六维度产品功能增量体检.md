## A. 数据质量与健壮性

### A-1 [P0] 字段缺失时显示 "null" / "NaN" / "undefined"

**现象**：`buildSummaryForDriver()` 直接拼接字段值：
```js
`${displayValue(driver.age)}岁`,  // age=null → "暂无数据岁"
`${displayValue(driver.consecutive_days)}天`
```

`displayValue()` 函数对 `null` 返回 `"暂无数据"`，但 `formatMetricValue()` 对 `NaN` 会原样返回字符串 `"NaN"`。更严重的是——如果 driver 对象本身缺少某个 schema 字段（compact 解码后该字段为 undefined），`displayValue(undefined)` 返回 `"暂无数据"` 拼接后变成 **"暂无数据岁"** 这种荒诞文案。

**实际触发场景**：
- 新上线城市/产品线的司机可能缺少某些字段
- 数据构建脚本 bug 导致某天某字段全量丢失
- strategyEvidence 索引解码越界 → evidence 为空对象 → 策略卡显示"暂无可展示依据"（客服完全不知道为什么）

**影响范围**：详情页综合画像、策略依据区、司机资料所有字段

**建议修复**：
1. `buildSummaryForDriver` 加字段存在性校验，缺字段跳过不拼接
2. `formatMetricValue` 对 NaN/Infinity 统一返回 `"暂无数据"`
3. evidence 解码失败时在客服参考区记录原始索引值（便于排查），主显区给降级文案
4. 构建脚本加 schema 完整性校验，输出字段覆盖率报告

### A-2 [P1] 筛选结果为 0 时无"调整建议"

**现象**：当前筛选无匹配时只显示一句 `"输入司机ID，或选择城市、公司、产品线、日期后显示匹配结果。"`。但客服已经选了筛选条件却得到 0 结果——**是条件太严了？还是这个城市今天确实没人？** 完全没有提示。

**客服真实痛点**："我选了柳州市+高风险+连续出车≥15天，结果是0。我是不是条件设错了？还是柳州真的没这类司机？"

**建议**：
- 0 结果时显示**最宽松条件的匹配数**（如"去掉连续出车条件后有 23 位匹配"）作为调整引导
- 显示当前各筛选项的独立命中数预览（如"城市柳州: 1,234位 / 高风险: 89位 / 连续出车≥15: 12位"）

### A-3 [P1] 列表卡片缺少"命中策略数"和"已联系状态"

**现象**：列表卡片 HTML（第 831 行）：
```js
item.innerHTML = `<strong>${driverId}</strong><span>${subtitle}<br/>风险分 ${riskTierScore} · 疲劳分 ${tiredScore}</span>`;
```

只有 ID + 副标题 + 两个裸分数。客服扫列表时看不到：
- 这位司机命中了几条策略（1条还是5条？优先级差很远）
- 是否已经被联系过
- 风险分/疲劳分的含义（27.44 是高是低？）

**建议**：卡片增加策略命中数 badge + 风险色标 + 已联系标记

### A-4 [P2] 无数据异常监控/告白机制

**现象**：daily 文件如果某天构建失败（行数为 0、schema 缺失、全部字段为 null），页面静默加载后显示空列表或报错，没有任何"数据异常"标识。

**建议**：加载完成后做基础健康检查（行数 > 0、schema 字段完整率 > 90%、核心指标非 null 率 > 95%），异常时在 meta 区域显示橙色警告图标 + 说明。

## B. 交互体验（客服日常操作痛点）

### B-1 [P1] 切换日期后选中状态和详情完全丢失

**现象**：正在查看某司机的详情页，切换日期后：
1. 详情页回到空白（"请选择或输入一个司机ID"）
2. 左侧列表重新渲染，之前的 active 高亮消失
3. 如果想看同一位司机在新日期的数据，需要**重新搜索该 ID**

**客服痛点**："我刚看完这位司机 07-06 的数据，想对比一下 07-07 有没有变化——切完日期人没了"

**建议**：切换日期时保留当前 driverId，自动尝试加载新日期的同 ID 司机；若不存在再回退到空白态。

### B-2 [P1] 无法在列表做快速标记/备注

**现象**：没有 checkbox 批量选择、没有右键菜单、没有"已联系"快捷标记。客服只能靠脑子记住谁联系过了。

**建议**（轻量方案）：
- 列表卡片加"已联系/待联系/已预约回访"三态切换按钮（localStorage 存）
- 支持批量勾选 + 一键标记为"今日已联系"
- 不碰后端，纯前端 localStorage

### B-3 [P1] 无键盘导航支持

**现象**：
- Tab 键无法跳到"建议策略"区域（strategy-list 没有 tabindex 或 heading 锚点）
- 方向键无法在司机列表上下切换
- Enter 键无法打开选中司机的详情

**对于每天要处理几十个司机的客服来说，键盘效率差距很大**

**建议**：
- 列表卡片增加 `tabindex="0"` + 上下方向键导航
- 建议策略区增加 `<h2>` 锚点 + `id="strategies"`，支持 URL hash 跳转
- 全局快捷键：`/` 聚焦搜索框，`Esc` 关闭详情

### B-4 [P2] 无搜索历史/最近查询

**现象**：每天上班第一件事就是重复设置相同的筛选条件（柳州市 + 高风险 + 产品线=专车）。没有"保存筛选"或"常用查询"功能。

**建议**：localStorage 存最近 5 套筛选条件，侧栏顶部展示"常用查询"快捷入口。

### B-5 [P2] 列表分页"继续显示50位"不够直观

**现象**：当前用"加载更多"按钮分页，每次追加 50 条。但不知道总共多少页、已加载多少比例。

**建议**：进度条或 "显示 150/320 位" 计数器；或者改为虚拟滚动（一次性渲染 DOM 少但可滚动浏览全部）。

## C. 业务逻辑深层缺陷

### C-1 [P0] 全局排名 vs 筛选内排名错位（排序失效）

**现象**：`filteredDrivers()` 第 805 行按 `riskTierRank` 升序排列：
```js
.sort((left, right) => {
    const leftRank = Number(left.riskTierRank);
    const rightRank = Number(right.riskTierRank);
    return (Number.isFinite(leftRank) ? leftRank : Infinity) -
           (Number.isFinite(rightRank) ? rightRank : Infinity);
})
```

`riskTierRank` 是**18 万+司机的全局排名**。当筛选到"柳州市·鱼峰区·专车"只剩 30 人时：
- 排名 943506 和 944000 的差异微乎其微（在全局视角下几乎相同）
- 但这 30 个人内部的风险差异可能巨大（一个 tiredScore=22 一个 tiredScore=5）

**结果**：**筛选范围越小，全局排名的区分度越低，排序越接近随机。** 客服看到的"第一个"不一定是该筛选条件下最该联系的司机。

**影响评估**：这是**当前网站最隐蔽也最严重的业务逻辑缺陷**——它让"识别谁先联系"这一步直接给出错误答案，且不会报错。

**建议修复**（二选一）：
1. **筛选内重排**：先按 riskTierRank 排，再用 tiredScore 作为同分段内的 secondary sort
2. **筛选内百分位**：在当前筛选集内计算每个司机的 tiredScore 百分位，用筛选内相对位置替代全局绝对排名

**推荐方案 1**，改动最小且效果明确。

### C-2 [P1] tiredScore 有值但未被任何业务逻辑消费

**现象**：
- `tiredScore` 在 daily 数据中存在（如 22.87）
- 列表卡片显示了裸数字（"疲劳分 22.87"）
- 详情页 header 显示了裸数字（"分数排名 943506"）
- **但没有一处用它做判断**：不做风险分级、不做排序阈值、不做关注入口

**22.87 对客服意味着什么？是高是低？跟谁比？完全没解释。**

**建议**：
- 列表卡片把裸数字替换为风险色标（基于 tiredScore 分段：绿<15 / 黄15-25 / 红>25）
- 详情页 header 的"分数排名"升级为"疲劳等级 + 同城相对位置"
- 用 tiredScore 做"今日 TOP 待联系"的入榜门槛

### C-3 [P1] "常规关怀"对所有未命中司机同一化

**现象**：fallbackRule 对所有未命中司机输出同一句：
```
"当前未命中明显风险策略，或策略计算结果暂不可用。
 可做常规状态确认，提醒司机保持安全驾驶、合理休息和规律作息。"
```

但实际上：
- **A司机**：tiredScore=3，所有指标都在 p25 以下，非常健康 → 应该说"这位师傅各项指标都不错，常规问候即可"
- **B司机**：tiredScore=21，只差 0.01 就触发 night-work 策略 → 应该说"接近夜间出车阈值，建议主动关怀"

两者都看到同样的"常规关怀"，**B 司机被遗漏的风险远大于 A 司机**。

**建议**：fallbackRule 增加"接近阈值"分级——对距阈值 <10% 的司机输出"重点关注"版常规关怀，对远离阈值的输出标准版。

### C-4 [P2] 策略优先级只有数字，无语义标签

**现象**：每条 rule 有 `priority` 数字（如 20、30、999），但渲染时只用它排序，**不在 UI 上体现主次关系**。客服看到 5 张平铺的策略卡，不知道哪张该先聊。

**建议**：priority 映射到语义标签 + 视觉权重：
- priority ≤ 20 → 标签"优先沟通"+ 卡片左侧红色竖线 + 稍大字号
- priority 21-50 → 标签"次要补充"+ 默认样式
- priority > 50 → 标签"参考信息"+ 折叠默认收起

## D. 性能与稳定性隐患

### D-1 [P0] 跨日期全量加载可能导致浏览器崩溃

**现象**：`ensureDriversForFilters()` 第 410-425 行：

```js
async function ensureDriversForFilters(filters) {
    // ...
    if (filters.dt) {
        await ensureDriversForDate(filters.dt);  // 选了日期：加载1天 ✅
        return;
    }
    const dates = driverDataIndex.dates || [];
    const all = [];
    for (const dataDate of dates) {  // 没选日期：加载全部天数！！
        await ensureDriversForDate(dataDate);
        all.push(...(loadedDriversByDate.get(dataDate) || []));
    }
    allDrivers = all;  // 7天 × 18万 ≈ 126万条记录
}
```

**当客服不选日期直接点查询时**：
- 系统会顺序加载全部 7 个 daily 文件
- 每个 ~54MB → 总计 ~378MB JSON 解析 + 内存占用
- 126 万条记录存入 `allDrivers` 数组
- 后续 `filteredDrivers()` 要遍历 126 万条做 filter + sort

**后果**：低端电脑（很多客服用的是配发的办公机）**浏览器直接卡死或 OOM 崩溃**。即使不崩溃，首次筛选等待时间可能超过 30 秒。

**影响评估**：这是一个**生产事故级隐患**——只要有人不选日期点查询就会触发。

**建议修复**：
1. **强制要求选择日期才能查询**：不选日期时禁用查询按钮，提示"请先选择数据日期"
2. 或者：不选日期时**仅加载最新 1 天**数据，UI 明确提示"未选择日期，默认使用最新一天数据"
3. 同时加前端保护：`allDrivers.length` 超过阈值时拒绝继续并提示

### D-2 [P1] 日期切换重复加载无缓存复用

**现象**：`ensureDriversForDate()` 第 328 行有 `loadedDriversByDate` Map 缓存检查——**代码层面有缓存**。但如果用户刷新页面（F5），localStorage 层面没有任何缓存，必须重新下载 54MB。

**建议**：考虑用 sessionStorage 存一份当天已加载日期的 drivers 子集（仅存筛选后的结果，不是全量 18万），避免刷新后重复加载。

### D-3 [P2] 单日文件加载失败无部分降级

**现象**：某一天的 daily JSON 如果 404 或格式损坏，`fetchJsonWithProgress` 直接 reject → `loadStaticData()` 整体 catch → 页面显示错误信息。**其他 6 天的数据也无法使用**。

**建议**：单日加载失败时记入错误日志，跳过该日期继续加载其余日期；manifest 或 UI 中标注"XX日期数据不可用"。

## E. 合规治理深化

### E-1 [P1] CSV 导出无审计痕迹

**现象**：`downloadCsv()` 生成的 CSV 只有 2 列（司机ID + 综合画像），文件名固定为 `司机画像_筛选结果.csv`，无任何审计信息。

**合规风险**：
- 不知道谁导出的、何时导出的
- 不知道导出时的筛选条件是什么
- 无法追踪敏感数据流向
- 截图/转发后无法溯源

**建议**：CSV 增加元信息行（前几行为注释）：
```csv
# 导出时间: 2026-07-09 14:30:00
# 操作人工号: CSxxxx（从登录态或本地配置读取）
# 筛选条件: 城市=柳州市, 产品线=专车, 日期=2026-07-08
# 记录数: 23
司机id,综合画像
...
```

同时文件名带时间戳：`司机画像_筛选结果_20260709_143000.csv`

### E-2 [P1] 无访问留痕（即使客户端日志）

**现象**：没有任何机制记录"谁看了哪个司机"。即使是 localStorage 里存一条简单的 `{driverId, viewedAt}` 日志也好——目前完全没有。

**合规风险**：发生隐私事件时无法回答"这个司机的画像被谁看过"

**建议**：localStorage 存访问日志（最近 100 条即可，自动轮转），格式：`{ts, driverId, action: 'view'|'export', filtersSummary}`

### E-3 [P2] 整页截图无水印保护

**现象**：之前文档提到了 driverId 脱敏，但**整页截图**仍然可以把完整画像（含所有指标数值）直接发给外部人员。

**建议**：CSS 背景层叠加半透明水印（工号 + 当前时间），打印/截图时自动带入。纯 CSS 实现：`body::after { content: "CSxxx · 2026-07-09 14:30"; ... }`

## F. 信息架构

### F-1 [P1] "综合画像"段落是不可扫描的文字墙

**现象**：`buildSummaryForDriver` 输出一大段逗号拼接文字：
```
55岁，柳州市，专车，xx公司，连续出车19天，当日服务8.32小时，夜间出车占比0.4148，睡眠不足0天，数据日期2026-07-08。 重点关注：夜间出车较多、连续出车偏多、高龄司机、服务时长偏高、空闲间隔偏多。
```

**问题**：
- 9 个指标 + 5 个策略标题挤成一段，**扫一眼抓不住重点**
- 数值精度不一致（"19天"整数、"8.32小时"两位小数、"0.4148"四位小数）
- 重点标签（"重点关注"后面的列表）混在末尾，容易被忽略

**建议**：改用结构化卡片布局（关键指标用色块 + 数值 + 趋势箭头），而非纯文本段落。

### F-2 [P1] "建议策略"区间视觉权重无差异

**现象**：所有策略卡用相同的 `<article class="strategy-card">` 渲染，priority=20 和 priority=99 的卡片**大小、颜色、位置完全一样**。唯一区别是 title 旁边的 tag 文字（"高风险"vs"常规"）。

**建议**：priority 映射到卡片视觉权重（见 C-4），高优卡片更大/更显眼，低优卡片可折叠。

### F-3 [P2] "司机资料"三组始终展开，长页面滚动累

**现象**：详情页三个 info-card（基础资料/出车指标/疲劳相关）**始终展开**，共约 15-20 个字段。如果司机命中多条策略 + 综合画像长，整个详情页可能需要滚 3-4 屏。

**建议**：默认折叠非关键分组（"司机资料"默认收起，点击展开）；或者把高频字段（年龄、城市、连续出车天数、夜间占比）提取到 header 区摘要中，减少展开"司机资料"的需求。

### F-4 [P2] 无"下一步行动"CTA

**现象**：客服看完一位司机的完整画像后，**页面上没有任何按钮告诉他接下来做什么**。没有"拨打电话"、"标记已联系"、"加入待回访"、"导出此司机"等行动入口。

**建议**：详情页底部固定 action bar（sticky footer），含 3-4 个核心操作按钮。

## G. 运营支撑

### G-1 [P1] 没有"我的工作台"视图

**现象**：网站只有"查询→看详情"的单次交互模式，没有聚合视角。客服无法回答：
- "我今天联系了多少人？"
- "还有多少人没联系？"
- "联系后的承诺兑现率如何？"（如果有记录的话）

**建议**：侧栏顶部或独立 tab 加"今日工作台"小面板（localStorage 数据驱动）：已联系 N / 待联系 M / 已预约回访 K。

### G-2 [P2] 无打印友好模式

**现象**：部分客服需要把司机画像打印出来归档或上报。当前页面的 CSS 打印效果未知（可能截断、溢出、背景色丢失）。

**建议**：添加 `@media print` 样式，隐藏侧栏筛选区、确保详情区完整打印、加上页眉页脚（含水印）。

### G-3 [P2] 无分享/转交备注功能

## 本次体检 TOP 3 最紧急项

### 🥇 D-1：跨日期全量加载可能导致浏览器崩溃
**严重度**：生产事故级（P0）  
**原因**：不选日期就查 → 加载 7×54MB = 378MB → 126 万条记录 → 浏览器 OOM  
**修复成本**：极低（加一行日期必选校验或默认最新 1 天）  
**建议**：**立即修复**——这是目前最容易触发且后果最严重的 bug

### 🥈 C-1：全局排名在筛选内排序失效
**严重度**：业务逻辑根本性缺陷（P0）  
**原因**：18 万人的全局排名在小样本筛选下失去区分度 → 客服看到的"第一位"未必是最该联系的  
**影响**：让"识别谁先联系"这一步给出错误答案  
**修复成本**：低（改一行 sort 函数加 secondary key）  
**建议**：**本周修复**

### 🥉 A-1：字段缺失时显示荒诞文案
**严重度**：数据健壮性（P0）  
**原因**：null/undefined/NaN 拼接后变成 "暂无数据岁" / "NaN"  
**影响**：降低客服对系统的信任度，尤其在数据异常期间  
**修复成本**：低（加几个 guard clause）  
**建议**：**本周修复**
