# 建议策略优先级与标签口径

更新时间：2026-07-09

## 1. 目标

建议策略模块用于帮助客服优先看到更值得关怀、也更容易解释给业务同学的风险线索。

排序不按页面命中顺序展示，而按“标签梯度 + 优先级”展示：

- 第一梯度：同时具备“高风险”和“可解释”的策略。
- 第二梯度：只有“高风险”的策略。
- 第三梯度：只有“可解释”的策略。
- 第四梯度：无标签策略。
- 同一梯度内部再按 `priority` 从小到大展示，通常体现为 P0 优先于 P1，P1 优先于 P2，P2 优先于 P3。

## 2. 标签定义

| 标签 | 样式 | 含义 |
|---|---|---|
| 高风险 | 红色标签 | case 中异常更显著，更适合作为优先关怀信号 |
| 可解释 | 黄色标签 | 业务、运营、安全同学容易理解为什么需要关怀 |

一条策略可以同时有两个标签，例如“连续高强度服务偏多”同时展示 `高风险` 和 `可解释`。

## 3. 当前核心字段优先级和替代字段

| 优先级 | 原字段 | 当前可用替代字段 | 标签 | 说明 |
|---|---|---|---|---|
| P0 | `consecutive_days_max_6h` | `consecutive_days_max`、`consecutive_days`、`is_long_consecutive` | 高风险 + 可解释 | 连续高强度工作，容易解释为疲劳累积 |
| P0 | `min_sleep_duration` | `sleep_deprivation_days`、`is_sleep_deprived` | 高风险 + 可解释 | 睡眠明显不足，健康和安全解释都强 |
| P0 | `peak_service_dur` | `server_dur_hour`、`online_dur_hour`、`lately_7d_except_sub_online_dur_hour` | 高风险 + 可解释 | 单日或高峰高强度服务，case 中区分明显 |
| P0 | `abnormal_driving_cnt` | `is_sudden_fatigue` | 高风险 + 可解释 | 当前只使用“是否突然累”做弱替代，不用疲劳分伪装异常驾驶次数 |
| P1 | `sleep_deprivation_days` | `sleep_deprivation_days`、`is_sleep_deprived` | 高风险 + 可解释 | 当前司机库已有，优先进入 P0 睡眠策略 |
| P1 | `temp_change_1d` | 暂无 | 高风险 + 可解释 | 当前司机库无天气字段，暂不触发 |
| P1 | `server_dur_sum_7d` | `lately_7d_except_sub_online_dur_hour`、`server_dur_hour_30d`、`lately_30d_online_dur_hour` | 高风险 | 近7天服务时长缺失时，用近7天在线和近30天服务/在线负荷替代 |
| P2 | `r_deep_night_peak` | `order_cnt_21_09_7d_rate`、`is_regular_night` | 可解释 | 深夜高峰字段缺失时，用夜间订单占比和常规夜班标签替代 |

策略优先使用原字段。当前每日司机库缺少原字段时，使用替代字段触发。后续每日文件带上原字段后，导入脚本会自动同时纳入原字段和替代字段。

## 4. 当前已补充的现有字段策略

用户补充的“现有字段里最值得用的高风险/解释字段”已按可解释性拆成独立策略，避免把弱替代字段伪装成原始异常驾驶次数：

| 策略 | 当前字段 | 优先级 | 标签 | 说明 |
|---|---|---|---|---|
| 疲劳评分偏高 | `tired_score`、`risk_tier_score`、`fatigue_index_7d`、`max_improved_tired_risk_score` | P1 | 高风险 + 可解释 | 使用 case 均值触发；其中 `max_improved_tired_risk_score` 为未来数据兼容字段，当前文件未必有值 |
| 身体指标需关怀 | `high_pressure`、`diastolic_bp_measure`、`systolic_bp_health`、`body_base_coeff`、`body_risk_factor`、`hyperglycemia_value`、`hyperlipidemia_value` | P2 | 可解释 | 当前表中 `body_risk_factor` 最接近的现有字段是 `body_base_coeff`（身体基础系数） |

未使用 `tired_score`、`risk_tier_score` 直接替代 `abnormal_driving_cnt`。原因是它们属于模型/评分信号，不是“异常驾驶次数”的原始行为计数；当前仅在独立的“疲劳评分偏高”策略中展示。

注意：这些新增字段已进入每日司机数据和策略配置，但仍受“死亡 case 参考样本 case_mean”约束。当前死亡 case 参考文件如果缺少 `tired_score`、`risk_tier_score`、`high_pressure`、`body_base_coeff` 等同名参考列，对应策略会记录有效样本数为 0，并按规则不触发。后续只要 case 参考文件补齐这些字段，构建脚本会自动计算 case 均值并开始触发。

## 5. 兼容保留策略

为了兼容当前已经存在的数据字段，同时补充以下策略：

| 策略 | 字段 | 优先级 | 标签 |
|---|---|---|---|
| 服务时长偏高 | `server_dur_hour` / `online_dur_hour` / `server_dur_hour_30d` | P1 | 高风险 + 可解释 |
| 连续出车偏多 | `consecutive_days` / `consecutive_days_max` / `is_long_consecutive` | P1 | 高风险 + 可解释 |
| 夜间出车较多 | `order_cnt_21_09_7d_rate` / `is_regular_night` | P2 | 可解释 |
| 高龄司机 | `age` | P3 | 可解释 |

## 6. 已删除策略

`past_7_day_non_listening_period` 对应的“近期空闲间隔偏多”已从建议策略中删除。

原因：

- 该字段更像出车节奏或在线行为指标，不是直接健康风险指标。
- 空闲间隔偏多可能来自休息、等单、离线、城市供需、个人事务等多种原因。
- 放在健康建议策略中不够稳定，客服容易追问“空闲多为什么风险高”。

该字段可以继续作为司机资料里的背景信息展示，但不再触发建议策略。

## 7. 阈值方向

策略仍由 `config/strategy-rules.json` 配置驱动，阈值来源仍为死亡 case 参考样本的 `case_mean`。

大部分字段使用：

```text
driver_metric > case_mean
```

`min_sleep_duration` 属于反向指标，睡眠越短风险越高，因此使用：

```text
driver_metric < case_mean
```

页面依据文案会根据配置里的 `operator` 自动展示“高于 case 均值”或“低于 case 均值”。

## 8. 话术转译 V1

建议策略卡片分为两层：

- 主显层：`对外话术`、`可操作建议`、`沟通提醒`，来自 `config/strategy-rules.json` 的 `translation` 配置。
- 参考层：`参考依据`，默认折叠，保留原内部依据和内部建议，供客服判断命中原因。

优先级标签按单个司机的命中策略列表动态兜底：如果当前司机没有任何策略被标成 `优先沟通`，页面会把排序第一条非常规策略提升为 `优先沟通`，确保客服始终有一个明确的首要沟通入口。该提升只影响前端展示，不改写 `config/strategy-rules.json` 中的原始 priority。

对外话术要求：

- 不出现 `case均值`、`死亡case`、`阈值`、`内部均值`。
- 不做医学诊断，不做事故因果判断。
- 优先使用关怀、确认状态、建议休息和调整出车节奏的表达。
- `复制话术` 只复制对外话术和可操作建议，不复制内部参考依据。

## 9. 发布包大小口径

CDN 一键部署包大小上限统一按 800MB 口径处理。每次更新数据或修改构建脚本后，相关文档、manifest 和验收口径都必须保持 800MB，不再使用旧的 500MB 说法。
