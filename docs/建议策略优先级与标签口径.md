# 建议策略优先级与标签口径

更新时间：2026-07-09

## 1. 目标

建议策略模块用于帮助客服优先看到更值得关怀、也更容易解释给业务同学的风险线索。

排序不按页面命中顺序展示，而按“优先级 + 分类标签”展示：

- P0 优先于 P1，P1 优先于 P2，P2 优先于 P3。
- 同优先级内，同时具备“高风险”和“可解释”的策略优先。
- 只有“高风险”的策略次之。
- 只有“可解释”的策略再次。
- 无标签策略最后展示。

## 2. 标签定义

| 标签 | 样式 | 含义 |
|---|---|---|
| 高风险 | 红色标签 | case 中异常更显著，更适合作为优先关怀信号 |
| 可解释 | 黄色标签 | 业务、运营、安全同学容易理解为什么需要关怀 |

一条策略可以同时有两个标签，例如“连续高强度服务偏多”同时展示 `高风险` 和 `可解释`。

## 3. 当前核心字段优先级

| 优先级 | 字段 | 中文含义 | 标签 | 说明 |
|---|---|---|---|---|
| P0 | `consecutive_days_max_6h` | 连续多天服务超过6小时的最大天数 | 高风险 + 可解释 | 连续高强度工作，容易解释为疲劳累积 |
| P0 | `min_sleep_duration` | 最短睡眠时长 | 高风险 + 可解释 | 睡眠明显不足，健康和安全解释都强 |
| P0 | `peak_service_dur` | 高峰期服务时长 | 高风险 + 可解释 | 单日或高峰高强度服务，case 中区分明显 |
| P0 | `abnormal_driving_cnt` | 异常驾驶次数 | 高风险 + 可解释 | 行为异常，和疲劳或身体状态异常容易关联 |
| P1 | `sleep_deprivation_days` | 睡眠不足天数 | 可解释 | 解释性强，但优先级低于最短睡眠时长 |
| P1 | `temp_change_1d` | 近1天气温变化幅度 | 高风险 + 可解释 | 天气突变叠加强劳动，对身体负担好解释 |
| P1 | `server_dur_sum_7d` | 近7天服务时长总和 | 高风险 | 和高峰服务时长组合后效果稳定 |
| P2 | `r_deep_night_peak` | 深夜高峰服务偏好/占比 | 可解释 | 深夜工作容易理解为疲劳风险 |

当前数据文件如果暂时没有这些字段，策略不会触发。后续每日文件带上字段后，导入脚本会按配置自动计算。

## 4. 兼容保留策略

为了兼容当前已经存在的数据字段，暂时保留以下策略，但展示优先级低于 P0 核心字段：

| 策略 | 字段 | 优先级 | 标签 |
|---|---|---|---|
| 服务时长偏高 | `server_dur_hour` / `server_dur_hour_30d` | P1 | 高风险 + 可解释 |
| 连续出车偏多 | `consecutive_days` | P1 | 高风险 + 可解释 |
| 夜间出车较多 | `order_cnt_21_09_7d_rate` | P2 | 可解释 |
| 高龄司机 | `age` | P3 | 可解释 |

## 5. 已删除策略

`past_7_day_non_listening_period` 对应的“近期空闲间隔偏多”已从建议策略中删除。

原因：

- 该字段更像出车节奏或在线行为指标，不是直接健康风险指标。
- 空闲间隔偏多可能来自休息、等单、离线、城市供需、个人事务等多种原因。
- 放在健康建议策略中不够稳定，客服容易追问“空闲多为什么风险高”。

该字段可以继续作为司机资料里的背景信息展示，但不再触发建议策略。

## 6. 阈值方向

策略仍由 `config/strategy-rules.json` 配置驱动，阈值来源仍为死亡 case 参考样本的 `case_mean`。

大部分字段使用：

```text
driver_metric > case_mean
```

`min_sleep_duration` 属于反向指标，睡眠越短风险越高，因此使用：

```text
driver_metric < case_mean
```

页面依据文案会根据配置里的 `operator` 自动展示“高于 case 均值”或“低于 case 均值”。

## 7. 发布包大小口径

CDN 一键部署包大小上限统一按 800MB 口径处理。每次更新数据或修改构建脚本后，相关文档、manifest 和验收口径都必须保持 800MB，不再使用旧的 500MB 说法。
