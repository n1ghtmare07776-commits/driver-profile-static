# 数据与接口契约

本次复刻可以采用纯静态 JSON，也可以采用本地后端 API。重点是前端功能和数据语义一致。

## 推荐字段

样例数据至少包含以下字段：

| 字段 | 中文含义 |
|---|---|
| `driver_id` | 司机 ID |
| `product_level2_name` | 产品线二级名称 |
| `age` | 年龄 |
| `resident_city_name` | 居住/常驻城市 |
| `city_level` | 城市等级 |
| `company_name` | 公司 |
| `consecutive_days` | 连续出车天数 |
| `server_dur_hour` | 当日服务时长 |
| `server_dur_sum_30d` | 近 30 日服务时长 |
| `order_cnt_21_09_7d_rate` | 夜间出车占比（0-1） |
| `sleep_deprivation_days` | 睡眠不足天数 |
| `is_organized` | 是否组织化 |
| `risk_tier_rank` | 源表模型排名 / 上游分层排名，不等同于当前静态站点人数 |
| `risk_tier_score` | 风险分 |
| `tired_score` | 疲劳分 |
| `dt` | 数据日期 |

可以额外保留旧表中的其他字段，用于详情卡片展示。

## 每日数据窗口

由于 CDN 一键部署包大小上限为 800MB，生产发布包不得持续累计历史数据库或历史司机明细。后续每次接收新一天数据时，默认只发布上传当天向前含当天的最新 7 个自然日数据。

示例：

```text
upload_date = 2026-07-06
window_start = 2026-06-30
window_end = 2026-07-06
```

所有筛选候选项、司机列表、司机详情、CSV 下载和建议策略结果都应基于这个窗口。静态 JSON 产物应额外生成 manifest，记录 `generated_at`、`upload_date`、`window_start`、`window_end`、`row_count`、`contains_queryable_driver_data`。

`generated_at` 必须由构建或上传脚本在生成静态站点产物时写入当前实时时间，不能手工写死。CDN 静态版前端左侧的 `静态站点生成时间` 应读取 `manifest.generated_at` 并格式化展示为 `YYYY/MM/DD HH:mm`。

页面顶部展示 `risk_tier_score` 作为综合风险参考；司机列表和高级筛选里的“近七天最高排名”使用 `bestRiskTierRank`，即前端在最近 7 天窗口内按司机 ID 去重后，取该司机命中过的最靠前 `risk_tier_rank`。`risk_tier_rank` 来自上游模型排名空间，可能大于当前静态站点司机总数。

## 静态 JSON 实现建议

静态 CDN 版采用按日独立的稳定静态版：

```text
data/drivers.json          最近 7 天窗口内的日期索引和每日文件清单
data/daily/drivers-YYYY-MM-DD.json
                           某一个自然日的页面可查询静态数据
data/filter-options.json   筛选候选项
data/manifest.json         生成时间、上传日期、数据窗口和条数
```

如果做 CDN 静态版，推荐构建脚本生成：

```text
dist/data/drivers.json
dist/data/daily/drivers-YYYY-MM-DD.json
dist/data/filter-options.json
dist/data/strategy-thresholds.json
dist/data/manifest.json
```

本需求包提供一键导入脚本：

```text
python3 scripts/import-driver-data.py data/incoming/司机库数据_20260706.xlsx --upload-date 2026-07-06 --out-dir dist/data
```

脚本会读取 Excel/CSV/JSON，过滤最新 7 天，生成稳定静态版 `drivers.json` 日期索引、按日独立的 `daily/drivers-YYYY-MM-DD.json`、筛选项、`meta.json`、`strategy-thresholds.json` 和 `manifest.json`。

`drivers.json` 只用于记录日期索引，不放多天司机明细。结构示例：

```json
{
  "mode": "daily-static-index",
  "latestDate": "2026-07-07",
  "dates": ["2026-07-03", "2026-07-07"],
  "files": [
    {
      "date": "2026-07-03",
      "path": "daily/drivers-2026-07-03.json",
      "row_count": 200000
    },
    {
      "date": "2026-07-07",
      "path": "daily/drivers-2026-07-07.json",
      "row_count": 165000
    }
  ]
}
```

`daily/drivers-YYYY-MM-DD.json` 用于筛选、列表、排序、CSV 下载和详情展示。为了让 20 万级单日数据仍能稳定控制大小，推荐使用紧凑结构：每条司机保留页面必需字段、综合画像摘要和已计算好的建议策略，前端点击司机时再把这些字段组装成详情卡片。不得包含其他日期生产司机明细或页面不需要的全量原始字段。结构示例：

```json
{
  "mode": "daily-static",
  "date": "2026-07-07",
  "drivers": [
    {
      "driverId": "100000000001",
      "city": "北京市",
      "company": "示例公司A",
      "product": "快车",
      "age": 56,
      "consecutive_days": 5,
      "server_dur_hour": 9.5,
      "server_dur_sum_30d": 180,
      "order_cnt_21_09_7d_rate": 0.42,
      "sleep_deprivation_days": 2,
      "isOrganized": "是",
      "riskTierRank": 128,
      "riskTierScore": 82.1,
      "tiredScore": 76.3,
      "dataDate": "2026-06-30",
      "subtitle": "北京市 · 快车 · 示例公司A",
      "summary": "56岁，北京市，快车，连续出车5天，当日服务9.5小时。",
      "strategies": [
        {
          "key": "regular-care",
          "title": "常规关怀",
          "evidence": "当前未命中明显风险策略。",
          "advice": "可做常规状态确认，提醒司机合理休息。"
        }
      ]
    }
  ]
}
```

前端点击司机时应直接使用该条司机记录中的字段和 `strategies` 组装详情，不再请求单司机详情 JSON，也不要求每条司机重复内嵌完整 `profile.groups` 模板。

## 后端 API 实现建议

如果使用后端 API，建议提供：

### `GET /api/meta`

返回数据概况。

```json
{
  "row_count": 5000,
  "data_dates": ["2026-06-30"],
  "upload_date": "2026-07-06",
  "window_start": "2026-06-30",
  "window_end": "2026-07-06"
}
```

### `GET /api/filter-options`

返回筛选候选项。

```json
{
  "cities": ["北京市", "上海市"],
  "companies": ["示例公司A", "示例公司B"],
  "products": ["快车", "专车"],
  "dates": ["2026-06-30"]
}
```

### `GET /api/drivers`

用于筛选司机列表。参数：

```text
driver_id
city
company
product
dt
age_min
age_max
consecutive_days_min
consecutive_days_max
server_dur_hour_min
server_dur_hour_max
order_cnt_21_09_7d_rate_min
order_cnt_21_09_7d_rate_max
sleep_deprivation_days_min
sleep_deprivation_days_max
is_organized
limit
```

注意 `city`、`company`、`product` 可以重复传多个值。

返回：

```json
{
  "drivers": [
    {
      "driverId": "100000000001",
      "subtitle": "北京市 · 快车 · 示例公司A",
      "riskTierScore": "82.1",
      "tiredScore": "76.3",
      "dataDate": "2026-06-30",
      "summary": "56岁，北京市，快车，连续出车5天，当日服务9.5小时。",
      "strategyTitles": ["服务时长偏高"]
    }
  ]
}
```

静态 CDN 版对应 `data/drivers.json` 日期索引和 `data/daily/drivers-YYYY-MM-DD.json` 按日明细，不需要后端 API。前端在加载首页时先读取日期索引，用户选择数据日期后再读取对应日期文件。

### `GET /api/drivers/{driver_id}`

返回单个司机详情。

```json
{
  "driverId": "100000000001",
  "summary": "56岁，北京市，快车，连续出车5天，当日服务9.5小时。",
  "meta": {
    "dataDate": "2026-06-30",
    "riskTierRank": "128"
  },
  "header": {
    "title": "司机画像",
    "chips": [
      {"kind": "region", "label": "北京市"},
      {"kind": "business", "label": "快车"},
      {"kind": "problem", "label": "连续出车"}
    ]
  },
  "strategies": [
    {
      "key": "service-duration-high",
      "title": "服务时长偏高",
      "evidence": "当日服务时长9.5小时，高于case均值5.5928小时；近30天服务时长180小时，高于case均值146.2457小时。",
      "advice": "建议关注司机服务时长是否连续偏高，沟通时提醒把休息排进出车计划。",
      "priority": 10
    }
  ],
  "groups": [
    {
      "key": "basic",
      "title": "基础资料",
      "items": [
        {"displayLabel": "年龄 age", "displayValue": "56"},
        {"displayLabel": "公司 company_name", "displayValue": "示例公司A"}
      ]
    }
  ]
}
```

静态 CDN 版的司机详情由对应日期的 `data/daily/drivers-YYYY-MM-DD.json` 中的司机字段、综合画像摘要和已计算好的 `strategies` 组装。前端点击司机或输入司机 ID 命中后，直接展示已加载的静态档案，不再请求单司机详情文件。

### `GET /api/strategies/thresholds`

可选。返回阈值说明，用于验收和排查，不应返回真实敏感参考样本。

```json
{
  "generatedAt": "2026-07-06T10:00:00+08:00",
  "rules": [
    {
      "key": "service-duration-high",
      "policy": "case_mean",
      "operator": ">",
      "thresholdValue": 8.6,
      "sampleCount": 120
    }
  ]
}
```

### `GET /api/drivers/export`

下载当前筛选结果 CSV。

CSV 至少包含：

```text
司机id,综合画像
100000000001,56岁，北京市，快车，连续出车5天，当日服务9.5小时。
```

## 排序

筛选结果优先按 `risk_tier_rank` 升序排序。如果为空，排在后面。

## 综合画像生成建议

综合画像是给客服快速扫读的短句。建议包含：

- 年龄
- 城市
- 产品线
- 公司
- 连续出车天数
- 服务时长
- 夜间出车占比
- 睡眠不足天数
- 是否组织化
- 数据日期

不要写成医学诊断或事故因果判断。

## 建议策略生成建议

建议策略流程：

```text
读取策略配置
  -> 读取或计算阈值
  -> 读取司机指标
  -> 执行比较逻辑
  -> 命中则生成 evidence 和 advice
  -> 无命中则返回常规关怀
  -> 按 priority 排序
```

策略规则建议使用 JSON 或 YAML 维护，至少包含 `key`、`enabled`、`title`、`priority`、`priority_tier`、`badges`、`threshold`、`evidence_template`、`advice_template`。当前生产阈值方法为 `case_mean`：大部分字段高于死亡 case 对应字段的 case 均值才触发展示；`min_sleep_duration` 等反向指标可通过配置使用低于 case 均值触发。后续如需更换算法，应只改配置和计算层，不修改页面渲染代码。

策略按 P0/P1/P2/P3 优先级展示，高风险标签为红色，可解释标签为黄色。`past_7_day_non_listening_period` 不再作为建议策略触发字段。完整口径见 [建议策略优先级与标签口径](建议策略优先级与标签口径.md)。
